# `grpc-monorepo-starter` — Implementation Plan

> A `degit`-able starter that gives any team a working gRPC monorepo in 60 seconds: pnpm workspaces, Turborepo, NestJS servers, `ts-proto` codegen, Buf for lint + breaking-change checks, plus Go and Python client gen out of the box. Public OSS, MIT.

**Status:** Draft — pending decisions in §11 before Phase 0 starts.

---

## 1. Goals & non-goals

### Goals
- A team runs `pnpm dlx degit mateokadiu/grpc-monorepo-starter my-platform`, `pnpm install`, `pnpm dev` and has two NestJS services talking gRPC on their laptop in under 60 seconds.
- `proto/` is the single source of truth. Edit a `.proto`, run `pnpm proto:gen`, get type-safe TS clients + servers + Go + Python clients in one shot.
- Buf wired for **lint** (style + correctness rules from day one) and **breaking-change detection** (CI blocks PRs that break wire compat against `main`).
- Two reference services — `service-a` (server) and `service-b` (caller) — both NestJS 11 + Fastify, both consuming the same generated package.
- Two non-TS clients — Go and Python — generated by `buf generate` and committed under `clients/` as ready-to-vendor modules.
- One GitHub Actions workflow: PRs run `buf lint` + `buf breaking` + typecheck + test; pushes to `main` run the full matrix.
- Docker Compose for local dev: each service on its own port, all share the generated proto package via the workspace.
- An optional Phase 6: Pulumi stack to deploy the whole thing to Oracle Cloud Always Free — same playbook the user runs on his other repos.
- **Public, OSS, MIT.** Lives on GitHub from day one. Versioned via git tags only — this is a template, not a published package.

### Non-goals (for v1)
- No gRPC-Web / Connect-Web / browser clients. Those need a different transport story (Envoy / `grpc-web` proxy) and are downstream concerns.
- No service mesh (Istio / Linkerd). The starter ships container-to-container gRPC over HTTP/2 — mesh is layered on later if you need it.
- No opinionated authn / authz. Interceptors are stubbed with a TODO and a comment pointing to the docs; bring your own JWT / mTLS / OAuth.
- No managed schema registry. Buf Schema Registry (BSR) is mentioned in §13 but not required — `proto/` in-repo is the contract.
- No multi-cluster federation, no envoy filters, no xDS. Plain client-side load balancing only.
- No alternative codegen frameworks. `ts-proto` is the only TS generator wired up. (Connect-ES / `grpc-tools` discussed in §11.)
- No Java / C# / Rust client templates. Go + Python are the two most-asked-for non-TS targets; the rest are easy to add later by the consumer.

## 2. The problem

Every team that ships gRPC builds the same boilerplate, badly, in their first sprint:

- Hand-rolled `protoc` invocations with flag drift between machines.
- A `proto/` folder with no linter, so style is whatever the first author typed.
- No breaking-change detection — a field gets renumbered, the next deploy silently corrupts data.
- NestJS gRPC examples in the docs that work for the toy case but don't show the workspace shape, the codegen step, or how to share generated types across services.
- One person writes the codegen script; six months later they leave; nobody else knows how to add a service.
- Go and Python clients are needed for "just this one consumer" — and now there are three codegen scripts that diverge.
- CI eventually catches one breaking change, two months after it shipped.

The fix is not a 4000-line doc. It's a repo you clone. **Productize it.**

The pieces all exist:
- `pnpm` workspaces + Turborepo solve orchestration.
- `ts-proto` is the de-facto TS generator.
- Buf solves lint + breaking + multi-language codegen.
- NestJS has first-class gRPC support via `@nestjs/microservices`.

But nobody has glued them together into a one-command starter. nx has gRPC samples but they're brittle and tied to nx generators. Buf has starter templates but they're not NestJS-aware. `ts-proto` is a dep, not a project shape. This starter is the missing glue.

## 3. Architecture

```
              ┌─────────────────────────────────┐
              │ proto/  (source of truth)       │
              │   orders.proto, shipping.proto  │
              │   buf.yaml + buf.gen.yaml       │
              └────────────────┬────────────────┘
                               │
        ┌─────────────┬────────┴────────┬──────────────┐
        │             │                 │              │
   buf generate  buf generate     buf generate    buf lint /
   (ts-proto)    (Go protoc)      (Python protoc) buf breaking
        ▼             ▼                 ▼              │
 ┌──────────────┐ ┌──────────┐  ┌──────────────┐      │
 │ packages/    │ │ clients/ │  │ clients/     │      │
 │ proto-gen/   │ │ go/pb/   │  │ python/pb/   │      │
 │ (workspace)  │ │ *.pb.go  │  │ *_pb2.py     │      │
 └──────┬───────┘ └──────────┘  └──────────────┘      │
        │                                              │
   ┌────┴────────────┐                                 │
   ▼                 ▼                                 │
 ┌─────────────┐  ┌─────────────┐                     │
 │ service-a   │  │ service-b   │                     │
 │ NestJS+gRPC │  │ NestJS HTTP │                     │
 │ @GrpcMethod │◄─┤ ClientGrpc  │                     │
 │ :3001/:50051│  │ :3002       │                     │
 └─────────────┘  └─────────────┘                     │
                                                       │
                  ┌────────────────────────────────────┘
                  ▼
       ┌──────────────────────────┐
       │ .github/workflows/ci.yml │
       │  buf lint + breaking +   │
       │  turbo build + test      │
       │  (required on PRs)       │
       └──────────────────────────┘
```

**The codegen contract:** `proto/` is the only place anyone edits. Every other artifact (`packages/proto-gen/src/generated/`, `clients/go/pb/`, `clients/python/pb/`) is produced by `buf generate` and **committed** so consumers can `degit` the repo and have it work without installing `buf` locally. CI re-runs `buf generate` on PRs and fails if the diff against the committed output is non-empty — drift detection without trust.

**The workspace contract:** `packages/proto-gen` exports a single barrel `index.ts` that re-exports every generated module. Apps import via `@repo/proto-gen` (or whatever scope the consumer picks during scaffold). The generated code is checked in; turbo orchestrates the regen on `proto:gen`.

## 4. Tech stack

| Layer | Choice | Why |
|---|---|---|
| Runtime | Node 22 LTS | Native `fetch`, stable WebStreams, mature ESM |
| Language | TypeScript 5.7+ | `--moduleResolution=bundler` everywhere |
| HTTP framework | NestJS 11 + Fastify adapter | First-class gRPC support via `@nestjs/microservices`; matches the user's daily stack |
| gRPC runtime | `@grpc/grpc-js` + `@grpc/proto-loader` | Pure JS, no native deps; works on every CI runner |
| TS codegen | `ts-proto` 2.7+ with `nestJs=true,addGrpcMetadata=true,esModuleInterop=true,stringEnums=true,outputServices=grpc-js` | Single most idiomatic NestJS-friendly TS generator |
| Proto tooling | `buf` CLI (lint + breaking + generate) | Replaces hand-rolled `protoc` scripts; multi-lang codegen for free |
| Lint rules | `buf.yaml` with `DEFAULT` lint preset + `FILE_LAYOUT,PACKAGE_DIRECTORY_MATCH` | Sane out-of-the-box; consumer can tighten later |
| Go codegen | `buf generate` with `buf.build/protocolbuffers/go` + `buf.build/grpc/go` | Modern Go protoc-gen plugins via Buf remote |
| Python codegen | `buf generate` with `buf.build/protocolbuffers/python` + `buf.build/grpc/python` | Same model, no local toolchain needed |
| Monorepo | `pnpm` workspaces + Turborepo 2.x | Workspace deps + cached `proto:gen` + parallel `dev` |
| Tests | Vitest 3 | Fast, ESM-native; one config per app |
| Container | Docker Compose v2 | Local-only; production uses Phase 6 Pulumi |
| CI | GitHub Actions | Required-status-checks story is clean |
| IaC (optional) | Pulumi → Oracle Cloud Always Free | Same $0/mo playbook as the user's other repos |
| License | MIT | |

Buf is on the critical path; we accept the dep because it pays for itself in lint + breaking + multi-lang codegen on the first PR.

## 5. Public surface

### 5.1 Consumer's first hour

```bash
# Scaffold a new repo
pnpm dlx degit mateokadiu/grpc-monorepo-starter my-platform
cd my-platform

# Install + first codegen run (postinstall hook)
pnpm install

# Bring it up
pnpm dev          # turbo dev — service-a + service-b both running
# In another terminal:
curl http://localhost:3002/orders/123    # service-b calls service-a over gRPC
```

That's the 60-second goal. Everything past this is the dev loop.

### 5.2 Edit-a-proto dev loop

```bash
# 1. Edit a proto file
$EDITOR proto/orders.proto

# 2. Lint locally (matches CI)
pnpm proto:lint     # buf lint

# 3. Check for breaking changes vs main
pnpm proto:breaking # buf breaking --against '.git#branch=main'

# 4. Regenerate TS + Go + Python
pnpm proto:gen      # buf generate

# 5. Typecheck — generated types flow through the workspace
pnpm typecheck

# 6. Run tests
pnpm test
```

`pnpm proto:gen` writes:
- `packages/proto-gen/src/generated/*.pb.ts`
- `clients/go/pb/*/*.pb.go`
- `clients/python/pb/*_pb2.py` + `*_pb2_grpc.py`

All three trees are committed. PR'ing without regenerating after a proto change makes CI fail loud (drift check), not silently.

### 5.3 Adding a new service

```bash
# Copy the template
cp -r apps/service-a apps/service-c
# Edit apps/service-c/package.json (name, port)
# Edit apps/service-c/src/main.ts (gRPC bind port)
# Add to docker-compose.yml
# Done.
```

A `pnpm dlx <template-cli> add-service <name>` codemod is a Phase 7+ stretch goal — not in v1.

### 5.4 Required env (per service)

```
# apps/service-a/.env — HTTP_PORT=3001 · GRPC_PORT=50051 · GRPC_HOST=0.0.0.0 · LOG_LEVEL=debug
# apps/service-b/.env — HTTP_PORT=3002 · SERVICE_A_GRPC_URL=localhost:50051 (docker: service-a:50051)
```

`.env.example` lives next to each service. Loaded via `@nestjs/config` with Zod validation.

## 7. Key flows

### 7.1 The proto-change workflow (the whole point of this repo)

```
developer:  edits proto/orders/v1/orders.proto  (renames a field, say)
             │
             ▼
pre-commit:  pnpm proto:lint  →  buf lint           (style)
             pnpm proto:gen   →  buf generate       (regen all 3 targets)
             pnpm typecheck   →  tsc -b             (broken TS callers surface here)
             │
             ▼
push to PR
             │
             ▼
.github/workflows/ci.yml:
  job: proto-lint           buf lint                              (fast, <5s)
  job: proto-breaking       buf breaking --against '.git#branch=main'  (FAILS on rename)
  job: codegen-drift        buf generate && git diff --exit-code  (FAILS if PR forgot regen)
  job: build                turbo build                           (tsc across workspace)
  job: test                 turbo test                            (vitest per app)
  job: go-build             cd clients/go && go build ./...
  job: python-import        cd clients/python && python -c 'import pb.orders.v1.orders_pb2'
             │
             ▼
required-status-checks gate the merge.
```

`buf breaking` is configured against the **default branch** of the repo. Field renames, type changes, removals, reserved-tag conflicts — all caught before merge. The consumer can relax rules per package in `buf.yaml` (`breaking.use: WIRE_COMPATIBLE` is the default; tighten to `FILE` if they don't want renames either).

### 7.2 Service startup (hybrid HTTP + gRPC)

NestJS supports hybrid apps natively. `service-a` boots both:

```ts
// apps/service-a/src/main.ts
const env = loadEnv();
const app = await NestFactory.create<NestFastifyApplication>(
  AppModule,
  new FastifyAdapter({ trustProxy: true }),
);
app.connectMicroservice<MicroserviceOptions>({
  transport: Transport.GRPC,
  options: {
    package: ['orders.v1', 'shipping.v1'],
    protoPath: [
      join(import.meta.dirname, '../../../proto/orders/v1/orders.proto'),
      join(import.meta.dirname, '../../../proto/shipping/v1/shipping.proto'),
    ],
    url: `${env.GRPC_HOST}:${env.GRPC_PORT}`,
    loader: { keepCase: false, longs: String, enums: String, defaults: true, oneofs: true },
  },
});
await app.startAllMicroservices();
await app.listen(env.HTTP_PORT, '0.0.0.0');
```

`service-b` boots HTTP only and gets a `ClientGrpc` injected via `ClientsModule.registerAsync`.

### 7.3 Client-side: typed wrapper around `ClientGrpc`

The default NestJS gRPC client returns RxJS Observables. Most app code wants Promises. The starter ships a thin typed wrapper:

```ts
// apps/service-b/src/orders-client/orders-client.service.ts
import { Inject, Injectable, OnModuleInit } from '@nestjs/common';
import type { ClientGrpc } from '@nestjs/microservices';
import { firstValueFrom } from 'rxjs';
import type { OrdersServiceClient } from '@repo/proto-gen';
import { ORDERS_SERVICE_NAME } from '@repo/proto-gen';

@Injectable()
export class OrdersClientService implements OnModuleInit {
  private client!: OrdersServiceClient;
  constructor(@Inject('ORDERS_PACKAGE') private readonly grpc: ClientGrpc) {}

  onModuleInit() {
    this.client = this.grpc.getService<OrdersServiceClient>(ORDERS_SERVICE_NAME);
  }

  getOrder(id: string) {
    return firstValueFrom(this.client.getOrder({ id }));
  }
}
```

`OrdersServiceClient` and `ORDERS_SERVICE_NAME` are emitted by `ts-proto` with `nestJs=true`. Type-safe at the call site; no `any` in the consumer's app code.

### 7.4 Postinstall codegen

`scripts/postinstall.mjs` runs on `pnpm install`. It checks whether the committed `packages/proto-gen/src/generated/` tree is newer than the newest `.proto`. If yes, skip. If no, run `buf generate`. This guarantees a fresh clone with no `buf` installed still boots — Buf has a `npm` package that ships the binary (`@bufbuild/buf`), used as a workspace devDep.

```jsonc
// package.json (root)
{
  "devDependencies": {
    "@bufbuild/buf": "^1.50.0",
    "turbo": "^2.5.0"
  },
  "scripts": {
    "postinstall": "node scripts/postinstall.mjs",
    "proto:lint": "buf lint",
    "proto:breaking": "buf breaking --against '.git#branch=main'",
    "proto:gen": "buf generate",
    "dev": "turbo dev",
    "build": "turbo build",
    "test": "turbo test",
    "typecheck": "turbo typecheck"
  }

```

## 8. Examples

### 8.1 Example `.proto` (ships in the template)

```proto
// proto/orders/v1/orders.proto
syntax = "proto3";
package orders.v1;
import "common/v1/types.proto";
option go_package = "github.com/mateokadiu/grpc-monorepo-starter/clients/go/pb/orders/v1;ordersv1";

service OrdersService {
  rpc GetOrder(GetOrderRequest) returns (GetOrderResponse);
  rpc ListOrders(ListOrdersRequest) returns (ListOrdersResponse);
  rpc CreateOrder(CreateOrderRequest) returns (CreateOrderResponse);
}

message GetOrderRequest    { string id = 1; }
message GetOrderResponse   { Order order = 1; }
message ListOrdersRequest  { int32 page_size = 1; string page_token = 2; }
message ListOrdersResponse { repeated Order orders = 1; string next_page_token = 2; }
message CreateOrderRequest { string customer_id = 1; repeated OrderItem items = 2; common.v1.Money total = 3; }
message CreateOrderResponse { Order order = 1; }

message Order {
  string id = 1;
  string customer_id = 2;
  repeated OrderItem items = 3;
  common.v1.Money total = 4;
  OrderStatus status = 5;
  int64 created_at_ms = 6;
}
message OrderItem { string sku = 1; int32 quantity = 2; common.v1.Money unit_price = 3; }

enum OrderStatus {
  ORDER_STATUS_UNSPECIFIED = 0;
  ORDER_STATUS_PENDING = 1;
  ORDER_STATUS_CONFIRMED = 2;
  ORDER_STATUS_SHIPPED = 3;
  ORDER_STATUS_DELIVERED = 4;
  ORDER_STATUS_CANCELED = 5;
}
```

### 8.2 Server: NestJS controller using the generated type

```ts
// apps/service-a/src/orders/orders.controller.ts
import { Controller } from '@nestjs/common';
import { GrpcMethod } from '@nestjs/microservices';
import type {
  GetOrderRequest, GetOrderResponse,
  CreateOrderRequest, CreateOrderResponse,
} from '@repo/proto-gen';
import { OrderStatus } from '@repo/proto-gen';
import { OrdersService } from './orders.service.js';

@Controller()
export class OrdersController {
  constructor(private readonly orders: OrdersService) {}

  @GrpcMethod('OrdersService', 'GetOrder')
  async getOrder(req: GetOrderRequest): Promise<GetOrderResponse> {
    const order = await this.orders.findById(req.id);
    return { order };
  }

  @GrpcMethod('OrdersService', 'CreateOrder')
  async createOrder(req: CreateOrderRequest): Promise<CreateOrderResponse> {
    const order = await this.orders.create({
      customerId: req.customerId,
      items: req.items,
      total: req.total,
      status: OrderStatus.ORDER_STATUS_PENDING,
    });
    return { order };
  }
}
```

### 8.3 Client: HTTP-to-gRPC fan-out

```ts
// apps/service-b/src/orders/orders.controller.ts
import { Controller, Get, Param, HttpException } from '@nestjs/common';
import { OrdersClientService } from '../orders-client/orders-client.service.js';

@Controller('orders')
export class OrdersController {
  constructor(private readonly ordersClient: OrdersClientService) {}

  @Get(':id')
  async get(@Param('id') id: string) {
    try {
      return await this.ordersClient.getOrder(id);
    } catch (err: any) {
      if (err?.code === 5 /* NOT_FOUND */) throw new HttpException('Order not found', 404);
      throw err;
    }
  }
}
```

### 8.4 Go example client

```go
// clients/go/example/main.go
package main

import (
  "context"
  "log"
  "time"

  ordersv1 "github.com/mateokadiu/grpc-monorepo-starter/clients/go/pb/orders/v1"
  "google.golang.org/grpc"
  "google.golang.org/grpc/credentials/insecure"
)

func main() {
  conn, err := grpc.NewClient("localhost:50051", grpc.WithTransportCredentials(insecure.NewCredentials()))
  if err != nil { log.Fatal(err) }
  defer conn.Close()

  client := ordersv1.NewOrdersServiceClient(conn)
  ctx, cancel := context.WithTimeout(context.Background(), 2*time.Second)
  defer cancel()

  resp, err := client.GetOrder(ctx, &ordersv1.GetOrderRequest{Id: "ord_demo_1"})
  if err != nil { log.Fatal(err) }
  log.Printf("order: %+v", resp.GetOrder())
}
```

## 9. Test strategy

### 9.1 Unit
- NestJS controllers and services tested with Vitest + plain mocks for the deps they take. No gRPC transport in unit tests.
- `packages/grpc-runtime/` helpers tested as pure functions where possible (option builders).

### 9.2 In-process gRPC integration
- Each server app gets one integration test that boots `app.connectMicroservice(...)` against an ephemeral port, then calls each RPC end-to-end with a `@grpc/grpc-js` client built from the generated stubs.
- Vitest `globalSetup` boots the app once per suite; tests share the channel.
- This catches contract mismatches that unit tests miss (e.g., `@GrpcMethod()` name typos, missing handlers).

### 9.3 Contract tests
- **Buf breaking** is the contract test for wire compat. Run in CI against `main`.
- **Pact** is *not* shipped in v1. Adding Pact for gRPC means `pact-jvm` or `protobuff-mocks`-style stubs; the lift is too much for a starter. Documented as a "see also" with a link, deferred to a follow-up project. (See §11.)

### 9.4 Codegen drift
- A dedicated CI job: `buf generate && git diff --exit-code -- packages/proto-gen/src/generated clients/go/pb clients/python/pb`. Non-empty diff fails the job. This catches PRs that change `.proto` without committing regenerated output — the most common new-team mistake.

### 9.5 CI matrix
- Node: 22.x only (LTS). Adding 20 is two lines if a consumer wants it.
- OS: `ubuntu-latest`. Mac/Windows are not in the matrix — too slow, too rarely the failure mode.
- Go: 1.23+ for the `clients/go` build step.
- Python: 3.12 for the `clients/python` import-smoke job.

## 11. Decisions to confirm before Phase 0

| # | Decision | Recommended | Alternative |
|---|---|---|---|
| 1 | Workspace package scope | **`@repo/*`** (placeholder; consumer rewrites via `scripts/degit-rename.mjs` on scaffold) | `@grpc-monorepo-starter/*`, `@<consumer-scope>/*` |
| 2 | TS gRPC codegen | **`ts-proto` with `nestJs=true,outputServices=grpc-js`** — closest to NestJS idioms, type-safe stubs, the user's existing pattern | `grpc-tools` (raw, no NestJS sugar); `@bufbuild/protoc-gen-es` + Connect (different transport story); `protobuf-ts` (similar but less NestJS adoption) |
| 3 | Proto tooling | **Buf CLI for everything** (lint + breaking + generate, Go + Python via remote plugins) | Raw `protoc` with hand-rolled scripts (matches the reference repo but loses lint + breaking); `protobufjs` (TS-only, no Go/Python) |
| 4 | HTTP framework | **NestJS 11 + Fastify** — first-class `@GrpcMethod()` decorators, matches the user's daily stack | Raw `@grpc/grpc-js` server (less DI sugar); Connect-Node (different ecosystem); Fastify alone with hand-rolled gRPC server |
| 5 | Test runner | **Vitest 3** | Jest (slower ESM story); `node:test` (no NestJS testing utilities) |
| 6 | Contract tests | **None in v1 — Buf breaking is the contract** | Pact (heavy lift for gRPC, doc-only stub for now); `protobuff-mocks` (immature) |
| 7 | Linter / formatter | **Biome** (single binary, no ESLint plugin sprawl) — matches the reference repo | ESLint + Prettier (familiar, but slower CI and more config) |
| 8 | Generated code: commit or gitignore? | **Commit** — keeps `degit` users zero-setup; CI's drift-check job catches forgetful PRs | gitignore + always-regen-on-postinstall (more elegant, but requires Buf to be installable on every CI runner / fresh clone) |
| 9 | Go module path | **`github.com/mateokadiu/grpc-monorepo-starter/clients/go`** as the template; consumer's `degit-rename.mjs` rewrites to their own | Leave as `example.com/...` placeholder forcing the consumer to edit manually |
| 10 | Phase 6 (Oracle Cloud deploy) | **Include** — same playbook as the user's other repos, no extra cost | Skip — split into a follow-up `grpc-monorepo-starter-deploy` repo |
| 11 | Repo name / GH slug | **`grpc-monorepo-starter`** — exact, searchable, says what it is | `nest-grpc-starter`, `ts-grpc-monorepo`, `proto-monorepo` |
| 12 | Versioning | **git tags only** (`v0.1.0` for the template milestone); no npm publish — this is a template | Publish a `create-grpc-monorepo` CLI to npm (Phase 7+ stretch) |

## 12. Out of scope (explicit so we don't drift)

- **gRPC-Web / Connect-Web / browser clients.** A different transport story (Envoy or `grpc-web` proxy), out of scope. Documented in §13 with pointers.
- **Service mesh, Envoy sidecars, xDS.** Layered on top of this if needed; not the starter's concern.
- **Authn / authz.** A `TODO` interceptor in `packages/grpc-runtime/` with a comment block — consumer plugs in JWT / mTLS / OAuth their way.
- **Observability stack** (OTel, Prometheus, Jaeger). A logging interceptor ships; tracing + metrics are a follow-up. Note in README.
- **Managed schema registry (Buf BSR).** `buf push` works against a BSR repo, but the starter doesn't assume one — `proto/` in-repo is the contract.
- **Multi-cluster federation, traffic splitting, canary releases.** Deploy story.
- **Non-Node / non-Go / non-Python clients** in v1. Java / C# / Rust / Ruby are documented as "add a `buf.gen.yaml` block" snippets in README, not committed templates.
- **Database / persistence / migrations.** The example `OrdersService` uses an in-memory `Map`. Bring your own ORM.
- **CLI scaffolder.** `degit` covers v1. A `create-grpc-monorepo` interactive CLI is a Phase 7+ stretch.
